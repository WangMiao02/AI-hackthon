# budget_manager — 预算管家

## 1. 执行入口判断

**首先判断触发来源，然后跳转对应分支：**

| 来源 | 判断方式 | 跳转 |
|------|---------|------|
| order skill 消费事件 | `trigger_type == "linked"` + 消费数据已传入 | → §3 自动记账 |
| 定时系统触发（日报） | `trigger_type == "scheduled"` + 用户偏好为「每天」 | → 加载 `references/weekly-report.md` §消费看板 |
| 定时系统触发（周报） | `trigger_type == "scheduled"` + 用户偏好为「每周一」 | → 加载 `references/weekly-report.md` |
| 定时系统触发（月报） | `trigger_type == "scheduled"` + 每月1号（固定，不受偏好影响） | → 加载 `references/monthly-report.md` |
| 超支预警触发 | `trigger_type == "threshold"` + 消费达阈值 | → 加载 `references/alert-flow.md` |
| 用户主动唤起 | 对话中含预算/花费/消费/记账等意图 | → §2 意图识别与分流 |
| 用户未完成配置 | 首次使用，预算未设置 | → 加载 `references/onboarding.md` |

> **消费事件联动**：每当 `order` skill 完成一笔支付，自动将 `order_type`（消费类目）、`amount`（金额）、`timestamp`（时间）、`merchant`（商家）推送给本 Skill，实现无感记账。

## 3. 自动记账

> **核心原则**：只记实际支付，不记规划。只有 `order` skill 产生真实 `order_id` 和支付流水的消费才计入。用户口头提及的计划一律不入账。

### 3.1 消费事件接收

每当 `order` skill 完成支付，自动接收以下数据并入账：

| 字段 | 来源 | 示例 |
|------|------|------|
| order_type | 上游 Skill 类型 | 外卖 / 打车 / 酒店 / 门票 / 药品 / 日用品 / 到餐 |
| amount | 实际支付金额 | ¥33.00 |
| coupon_deduction | 优惠券抵扣 | -¥5.00 |
| merchant | 商家名称 | 老乡鸡（静安寺店） |
| timestamp | 支付时间 | 2025-05-20 12:03 |
| order_id | 订单号 | 用于去重和溯源 |

### 3.2 消费类目体系

| 类目 | 包含场景 | 对应上游 Skill |
|------|---------|---------------|
| 🍱 外卖 | 外卖点餐、拼单 | food_recommend, group_order |
| 🍽 到餐 | 堂食、排队取号后消费 | queue_number |
| 🚗 交通 | 打车、高铁、机票 | trip_reserver |
| 🏨 住宿 | 酒店预订 | hotel_reserver |
| 🎫 门票 | 演出、景区、展览 | concert_reserver / attraction_ticket / movie_ticket |
| 💊 医药 | 买药 | medicine_reminder |
| 🧴 日用品 | 家居消耗品 | supplies_reminder |
| 📦 其他 | 未归类消费 | — |

### 3.3 入账处理

1. 按 `order_id` 去重，防止重复记录
2. 按 `order_type` 自动归入对应消费类目（参照 §3.2）
3. 更新当月累计消费和各类目累计
4. **检查阈值**（按以下顺序依次执行）：
   - 遍历用户设定的所有分类目预警节点（默认 90%），检查该类目是否首次到达某节点且尚无已推标记 → 若命中，加载 `references/alert-flow.md` 推送分类目预警，写入该节点已推标记
   - 遍历用户设定的所有总预算预警节点（默认 70% / 100%），检查总消费是否首次到达某节点且尚无已推标记 → 若命中，加载 `references/alert-flow.md` 推送总预算预警，写入该节点已推标记
   - 同一笔消费同时命中分类目预警和总预算预警时：**先推分类目，再推总预算**
   - 若本次入账为退款（`refund_amount` 字段存在），检查退款后消费是否降回某已推节点阈值以下 → 若是，清除该节点已推标记，并按用户入账提醒偏好决定是否展示退款恢复提示
5. 入账静默执行，**不打扰用户**（除非触发预警或用户偏好为「每次都说」）

### 3.4 入账确认（按用户偏好决定）

入账提醒行为完全由 Onboarding 步骤二中用户设置的偏好决定：

- 用户选「每次都说」→ 每笔入账后展示以下确认：

```
✅ 已记录：外卖 ¥33.00（老乡鸡）

本月外卖已花 ¥847 / 预算 ¥1,500
```

- 用户选「不用，静默就好」→ 全程静默，不展示任何入账提示

## 4. 边界处理速查

| 场景 | 处理 |
|------|------|
| 未完成 Onboarding | 加载 `references/onboarding.md` 引导，完成前仅静默记账不推报告 |
| 当月无消费记录 | 周报/月报提示「本周/本月暂无消费记录」，不推空报告 |
| 单笔异常大额消费 | 入账后主动确认「检测到大额消费 ¥X，确认记录？」 |
| 退款/取消订单 | 从消费记录中扣减，更新累计（需 order skill 推送退款事件）|
| 跨月消费（月底下单次月到账）| 按支付完成时间归入对应月份 |
| 预算为 0 的类目 | 不做该类目预警，仅纳入总预算计算 |
| 优惠券抵扣 | 记录实际支付金额（已扣除优惠），优惠金额单独标注 |
| 用户要求删除数据 | 提供「清除全部消费数据」选项，不可恢复 |
| 多人共用账号 | 不支持，按单一用户维度记录 |

## 5. 数据存储与隐私

| 数据类型 | 存储方式 | 说明 |
|---------|---------|------|
| 预算配置 | 本地加密存储 | 总预算 + 各类目预算 + 预警阈值 |
| 消费流水 | 本地加密存储 | 仅保留最近 6 个月，更早自动归档为月度汇总 |
| 月度汇总 | 本地加密存储 | 各月总消费 + 类目分布（永久保留） |
| 洞察缓存 | 内存临时 | 分析结果缓存 24 小时 |

所有消费数据仅存储在用户本地设备，不上传消费明细到云端，分析计算全部在端侧完成，用户可随时导出或删除全部数据。

## 6. 本期不做（Backlog）

> 以下功能**不在当前 Skill 范围内**，遇到请告知用户暂不支持：

- 线下消费手动记账（仅覆盖美团生态内自动记录）
- 多账户 / 家庭共享预算
- 投资理财建议
- 信用卡账单导入
- 跨平台消费聚合（支付宝 / 微信等）
- 自动转账 / 存钱罐功能
- 年度预算规划（当前仅支持月度）
- 税务相关计算
- 消费记录导出为 Excel
