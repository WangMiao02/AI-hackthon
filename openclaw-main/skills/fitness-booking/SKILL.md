# fitness_booking — 运动健身预约

## 1. 执行入口判断

**首先判断触发来源，跳转对应分支：**

| 来源 | 判断方式 | 跳转 |
|------|---------|------|
| 定时系统触发（周期复约/打卡提醒） | `trigger_type == "scheduled"` + 周期提醒任务存在 | → §5 周期复约与打卡提醒 |
| 用户表达运动目标/计划 | 对话中含目标类信号（见 §2.1 目标引导触发词表） | → §2.1 目标引导分支 |
| 用户主动唤起 | 对话中含健身/运动/课程等明确意图 | → §2.2 意图识别与分流 |
| 用户未完成配置 | 首次使用，常用地址未设置 | → §6 Onboarding |

## 2. 意图识别与分流

### 2.1 目标引导分支

**触发词表**（模糊匹配，不要求完全一致）：

| 目标类型 | 示例表达 |
|---------|--------|
| 减脂塑形 | "想减肥"、"想瘦"、"练马甲线"、"减脂"、"塑形" |
| 增肌力量 | "想练肌肉"、"增肌"、"练力量"、"想变壮" |
| 放松舒缓 | "想放松"、"减压"、"拉伸"、"身体很僵" |
| 开始健身 | "想开始健身"、"想坚持运动"、"想养成运动习惯" |
| 特定运动 | "想学游泳"、"想试试瑜伽"、"想上搏击课" |

检测到以上信号后，**先做目标引导，再推荐合适的运动类型**：

```
听起来你想开始运动，太棒了！
你的目标是哪个方向？

[减脂塑形]  [增肌力量]  [放松拉伸]  [随便动动]
```

目标 → 推荐运动类型映射：

| 目标 | 推荐运动类型 |
|------|------------|
| 减脂塑形 | 动感单车、搏击操、HIIT团课、游泳 |
| 增肌力量 | 私教（力量方向）、健身房自由训练 |
| 放松拉伸 | 瑜伽、普拉提、拉伸课 |
| 随便动动 | 综合推荐（按距离+评分排序） |

用户选择目标后 → 展示对应运动类型供选择 → §2.2 意图识别与分流

### 2.2 服务意图识别

| 用户意图 | 示例表达 | 服务类型 | 跳转 |
|---------|---------|---------|------|
| 私教课 | "约个私教"、"想找教练"、"一对一训练" | `personal_training` | → §3 门店查询 |
| 瑜伽 | "上瑜伽课"、"瑜伽团课"、"想练瑜伽" | `yoga` | → §3 门店查询 |
| 普拉提 | "普拉提"、"想试试普拉提" | `pilates` | → §3 门店查询 |
| 搏击/格斗 | "搏击操"、"拳击课"、"格斗" | `boxing` | → §3 门店查询 |
| 动感单车 | "动感单车"、"spinning" | `cycling` | → §3 门店查询 |
| 游泳 | "游泳"、"约个泳道"、"游泳馆" | `swimming` | → §3 门店查询 |
| 健身房自由训练 | "去健身房"、"撸铁"、"自由训练" | `gym` | → §3 门店查询 |
| 意图不明确 | "帮我约个运动" | — | 追问：「想上团课、约私教、还是去健身房自由训练？」 |

> **私教特殊处理**：意图识别为 `personal_training` 时，额外询问训练方向（减脂/增肌/体能/康复），作为教练推荐的筛选条件。

## 3. 门店查询与推荐

**输入**：服务类型 + 用户地址（GPS/常用地址）+ 训练方向（如有）

**执行步骤**：

1. 调用美团运动健身接口，查询附近门店及课程
2. 按以下规则排序，取前 **2 家** 展示，超出折叠：

   | 优先级 | 筛选维度 | 说明 |
   |--------|---------|------|
   | 1 | 历史消费过该门店 | 熟悉环境，用户信任度高 |
   | 2 | 评分 | 4.8分及以上优先 |
   | 3 | 距离 | 同等条件取距离最近 |
   | 4 | 价格 | 同等条件取性价比最高 |

   **冷启动处理**（无历史消费记录）：按综合评分从高到低排列，展示评分 + 服务次数 + 人均价格

3. 无可用门店 → 提示「附近暂无匹配的健身场所，请尝试调整位置或运动类型」
4. 有结果 → 跳转 §4 预约确认

---

## 4. 预约确认

**执行步骤**：

1. 展示门店推荐卡片（含评分、人均价格、距离）
2. 用户选择门店 → 展示该店可选课程/项目（**含时长与价格**）
3. 私教课：展示可选教练（含评分、擅长方向、资质）；团课：展示可选班次（含剩余名额）
4. 用户选择教练/班次 → 展示可选时段
5. 用户选择时段 → 展示预约确认卡片
6. 用户确认 → 调用 `order` skill 完成支付 → §4.1 预约结果处理

卡片格式及教练/班次/时段交互详见 [references/booking-flow.md](references/booking-flow.md)

### 4.1 预约结果处理

**成功**：
```
✅ 预约成功！
🏋️ [课程/项目名称]（[时长]min） · [门店名称]
👤 教练：[教练姓名] ⭐[评分]   （私教课显示；团课显示班次名称）
📅 [预约日期] [时间段]
📍 [门店地址]
💰 ¥[金额]
📌 已写入今日行程，明早早报会提醒你出发时间。
```
→ 将预约信息写入当日行程，供 `morning_briefing` 早报「今日行程」模块读取，统一处理出发提醒
→ 预约成功后轻量询问是否设置健身计划：「要帮你设置定期健身提醒吗？」，用户说「好」才进入 §5.1

**失败**：
→ 提示失败原因，保留预约信息，引导重试或换门店/时段

## 5. 周期复约与打卡提醒

### 5.1 健身计划设置（用户主动触发）

用户说「帮我设置健身提醒」「每周提醒我去健身」等时触发，询问：

```
🏃 好的！你想多久运动一次？

[每周1次]  [每周2次]  [每周3次]  [自定义]
```

用户选择频率后，记录以下信息用于后续提醒：
- 运动类型（`yoga` / `personal_training` / `gym` 等）
- 上次门店信息（优先复用）
- 上次教练/班次信息（优先复用）
- 运动频率（每周 N 次）
- 下次提醒时间 = 当前日期 + 间隔天数 - 1天

### 5.2 到期提醒推送（定时触发）

到期前一天推送复约卡片，格式及交互详见 [references/periodic-reminder.md](references/periodic-reminder.md)

## 7. 边界处理速查

| 场景 | 处理 |
|------|------|
| 附近无可用门店 | 提示调整位置或扩大搜索范围 |
| 团课名额已满 | 提示「该班次已满」，推荐同类型其他班次或时段 |
| 用户选择时段已被预约 | 提示「该时段刚被预约」，引导重新选择 |
| 支付失败 | 提示原因，保留预约信息可重试 |
| 门店临时闭店 | 推送通知，引导重新选择门店 |
| 常用地址未设置 | 触发 §6 Onboarding |
| GPS 未授权 | 降级使用 Onboarding 中的常用地址 |
| 用户提及医疗康复/物理治疗 | 明确告知「医疗康复训练暂不支持，建议前往正规医疗机构或康复中心」 |
| 周期提醒到期但用户已手动预约 | 检测到近期已有预约记录则静默跳过 |
| 私教课无匹配方向教练 | 展示综合评分最高的教练列表，标注「可与教练沟通训练方向」 |
| 目标引导后用户不想预约 | 回应「好的，想运动了随时来找我～」，不再追问 |
