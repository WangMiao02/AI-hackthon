# hotel_reserver — 订酒店

## 1. 执行入口判断

**首先判断触发来源，跳转对应分支：**

| 来源 | 判断方式 | 跳转 |
|------|---------|------|
| trip_reserver 联动触发 | `trigger_type == "linked"` + 预订成功事件 | → §2 协议酒店检查 |
| 用户主动唤起 | 对话中含酒店/住宿/订房等意图 | → §2 协议酒店检查 |
| 用户未完成配置 | Onboarding 信息缺失 | → §5 Onboarding |

> **联动触发时**：从 `trip_reserver` 上下文中读取 `destination_city`（目的地城市）和 `travel_date`（出行日期）作为查询入参，无需用户重复填写。

---

## 2. 协议酒店检查

1. 检查用户是否开启「公司协议酒店」开关
2. 如果**已开启** → 静默跳过，不推送，Skill 结束
3. 如果**未开启** → 读取 Onboarding 偏好配置，跳转 §3

---

## 3. 查询与推荐

**输入**：目的地城市 + 出行日期 + 用户偏好（办公地址、价位区间、房型）

**执行步骤**：
1. 查询目的地酒店实时数据（房价、库存、距办公地距离）
2. 按以下规则排序，取前 **2 条** 展示，超出折叠：

   | 优先级 | 筛选维度 |
   |--------|---------|
   | 1 | 距用户常去办公地址最近 |
   | 2 | 价格落在用户预算区间内 |
   | 3 | 有用户偏好房型（大床 / 双床） |
   | 4 | 同等条件取评分最高 |

3. 如推荐酒店无偏好房型 → 推送时注明「暂无大床房，已推荐其他房型」
4. 如推荐价格超出预算 → 推送时注明超出幅度，由用户决策
5. 如**无匹配酒店** → 提示「暂无符合偏好的酒店」，引导用户自选，Skill 结束
6. 有匹配结果 → 跳转 §4 推送卡片

---

**成功**：

**失败**：
→ 提示失败原因，保留当前预订信息，引导用户重试或换一家

## 5. Onboarding（首次配置）

引导用户依次填写：

| 配置项 | 示例 |
|--------|------|
| 常去办公地址（目的地城市） | 上海市静安区南京西路1515号 |
| 价位偏好（单晚预算区间） | ¥300–600 |
| 房型偏好 | 大床 / 双床 / 无偏好 |
| 公司协议酒店 | 开 / 关（默认关） |

> 配置完成后自动跳转 §3 执行。已配置用户可在「差旅偏好设置」中修改。

---

## 6. 边界处理速查

| 场景 | 处理 |
|------|------|
| 用户开启协议酒店开关 | 静默跳过，不推送 |
| 推荐酒店无偏好房型 | 推送时注明，由用户决策 |
| 推荐价格超出预算 | 推送时注明超出幅度，由用户决策 |
| 所选酒店已满房 | 提示已满房，推荐次优选项 |
| 支付失败 | 提示原因，保留信息可重试 |

## 7. 本期不做（Backlog）

> 以下功能**不在当前 Skill 范围内**，遇到请告知用户暂不支持：

- 独立触发的酒店搜索（与交通票解耦）
- 退订 / 改期流程
- 公司协议酒店名单对接（自动匹配协议价）
- 多晚连续入住价格优化
- 出行当天 check-in 时间提醒
