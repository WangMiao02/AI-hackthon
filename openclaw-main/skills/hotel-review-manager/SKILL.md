# hotel_review_manager — 房客评价管理

## 1. 执行入口判断

| 来源 | 判断方式 | 跳转 |
|------|---------|------|
| 新评价事件 | 平台 API 推送新评价通知（房客退房后可触发） | → §2 评价分流 |
| 每日汇总写入 | `trigger_type == "daily_summary"`（固定时间触发） | → §5 日报写入 |
| 配置缺失 | 首次使用或配置不完整 | → §6 Onboarding |

> **核心设计原则**：Skill 由新评价事件驱动，无需商家主动唤起。
> 好评全自动处理，差评推卡片等待商家确认，商家仅需处理差评部分。

---

## 2. 评价分流

收到新评价后，按星级进入对应处理路径：

```
1-3星（差评）  →  §3 差评处理
4-5星（好评）  →  §4 好评处理
```

> 民宿平台满分通常为5星，3星及以下视为需要处理的差评。

---

## 3. 差评处理

### 3.1 投诉类型识别

读取评价内容，优先识别以下投诉类型（可多类型并存）：

| 投诉类型 | 典型关键词 | 下游动作 |
|----------|-----------|---------|
| 硬件问题 | 空调/热水/网络/门锁/电视/噪音/隔音 | 回复策略匹配 + 触发 `maintenance_reminder` |
| 卫生问题 | 脏/头发/异味/床单/毛巾/蟑螂/虫 | 回复策略匹配 + 触发 `maintenance_reminder` |
| 服务问题 | 态度/回复慢/入住流程/沟通/等待 | 回复策略匹配 |
| 与实际不符 | 照片/描述/面积/位置/景观 | 回复策略匹配 + 差评卡片「申请客服介入」按钮高亮 |
| 其他/泛差评 | 无明确投诉点 | 通用回复策略 |

### 3.2 AI 生成回复草稿

结合以下维度生成回复草稿：

- 识别出的投诉类型（见 §3.1 策略匹配）
- 商家已配置的品牌语气（见 §6 Onboarding）
- 历史差评回复风格（若有）

**各类型回复策略要点**：

| 投诉类型 | 回复策略要点 |
|----------|------------|
| 硬件问题 | 明确承诺"已记录，会在下次入住前修复"，避免泛泛道歉 |
| 卫生问题 | 表达重视 + 说明整改措施，避免"我们已清洁"等无实质内容的回复 |
| 服务问题 | 态度诚恳 + 具体改进承诺 |
| 与实际不符 | 温和说明实际情况，不与客人正面争论 |
| 其他/泛差评 | 通用致歉 + 欢迎再次入住 |

### 3.3 推送差评处理卡片

```
┌──────────────────────────────────────┐
│ 🔴 新差评  ⭐⭐☆☆☆  刚刚             │
│ 🔧 硬件问题                          │
│                                      │
│ 「空调完全不制冷，跟前台反映了也没   │
│   人来修，睡了一晚上热死了」         │
│                                      │
│ ────────────────────────────────── │
│ 💬 AI 建议回复                       │
│                                      │
│ 您好，非常抱歉空调故障影响了您的入   │
│ 住体验，且未能及时响应您的反馈，这   │
│ 是我们的失职。该问题已记录，会在下   │
│ 次有客人入住前完成维修。诚邀您下次   │
│ 再来，给您一个更好的体验。           │
│                                      │
│  [直接发送]  [修改后发送]  [暂不回复]│
│                                      │
│  [申请客服介入 →]                    │
└──────────────────────────────────────┘
```

**按钮响应**：
- `直接发送` → 调用 API 发送草稿内容，推送发送成功卡片
- `修改后发送` → 展开文本编辑区，商家修改后点击「确认发送」
- `暂不回复` → 关闭卡片，计入「待处理差评」，可在每日汇总卡片中查看
- `申请客服介入 →` → 跳转 `hotel_review_appeal`，传入当前评价内容；「与实际不符」类型此按钮高亮显示

> **硬件/卫生类差评**：卡片推送的同时，向 `maintenance_reminder` 抛出触发事件（含房间号、投诉类型、评价内容摘要），不在本卡片内展示维修流程。

## 4. 好评处理

### 4.1 自动生成并发送回复

读取评价内容，从商家预设好评回复模板库中匹配最合适的模板，
自动调用 API 发送，全程无需商家介入。

> 模板匹配维度：提及房间/设施 / 提及位置/景观 / 提及服务/沟通 / 纯泛好评 / 含晒图
> 同一用户不重复使用相同模板，避免机械感。

### 4.2 好评回复不单独推送通知

好评回复静默完成，仅在每日汇总卡片中呈现当日好评回复总数。

---

## 5. 日报写入

每日固定时间，将以下汇总数据写入 `hotel_daily_report`：

| 数据项 | 说明 |
|--------|------|
| 当日新增差评数 | 1-3星评价总数 |
| 差评已回复数 | 商家确认发送的回复数 |
| 差评待处理数 | 点击「暂不回复」或未响应的差评数 |
| 当日新增好评数 | 4-5星评价总数 |
| 好评自动回复数 | 系统自动发送的好评回复数 |
| 硬件/卫生类差评数 | 已向 maintenance_reminder 触发事件的差评数 |

> 写入格式与口径以 `hotel_daily_report` §3 数据项定义为准。

## 7. 边界处理速查

| 场景 | 处理方式 |
|------|---------|
| API 读取评价失败 | 推送失败提示卡片，附 [手动前往榛果商家版] 按钮 |
| API 发送回复失败 | 自动重试 1 次；仍失败则推送失败卡片，草稿内容保留可重新发送 |
| 差评 3 天内未响应 | 推送「有差评待处理」提醒卡片，附差评列表 |
| maintenance_reminder 触发失败 | 仅记录日志，不影响评价回复流程，下次日报写入时补触发 |
| 好评模板全部轮转完毕 | 从头轮转，同一用户不连续重复同一模板 |
| 商家修改回复内容为空 | 不允许发送，提示「回复内容不能为空」 |
