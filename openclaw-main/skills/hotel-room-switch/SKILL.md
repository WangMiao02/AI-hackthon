# hotel_room_switch — 房间上下架管理

## 1. 执行入口判断

| 来源 | 判断方式 | 跳转 |
|------|---------|------|
| 后台定时检测 | `trigger_type == "scheduled_check"`（每分钟执行） | → §2 房态检测 |
| 订单取消事件 | 平台 API 推送订单取消通知 | → §4 取消订单上架流程 |
| 退房事件 | 平台 API 推送退房通知 | → §5 退房上架流程 |
| 商家主动唤起 | 商家在对话中提及上架/下架/暂停接单/恢复接单等意图 | → §3 前置检查 |
| 配置缺失 | 首次使用或配置不完整 | → §6 Onboarding |

---

## 2. 房态检测（每分钟后台静默执行）

### 2.1 按房间类型分别检测

对每个房间类型独立执行以下判断：

```
读取该房型当前可售房间数
↓
可售房间数 ≤ 触发阈值（默认 0 间）
↓
是 → 该房型已下架？
      已下架 → 跳过，不重复操作
      未下架 → § 2.2 执行自动下架
否 → 跳过，本次检测结束
```

> 各房型独立判断，互不影响。
> 标准间满房不影响大床房的上架状态。

### 2.2 执行自动下架

1. 调用平台 API，将该房型下架（停止接单）
2. 记录下架时间戳、房型名称、触发时可售房间数
3. 推送下架通知卡片：

```
┌──────────────────────────────────────┐
│ 🔴 已自动下架                        │
│                                      │
│ 房型     大床房                      │
│ 原因     可售房间数已达触发阈值（0间）│
│ 下架时间  05月25日 14:32             │
│                                      │
│           [知道了]                   │
└──────────────────────────────────────┘
```

## 3. 前置检查（商家主动唤起时执行）

1. 读取商家配置，**配置缺失** → 跳转 §6 Onboarding，Skill 结束
2. 识别商家意图：

| 意图 | 跳转 |
|------|------|
| 下架 / 暂停接单 | → §3.1 主动下架 |
| 上架 / 恢复接单 | → §3.2 主动上架 |
| 意图不明确 | → 推房型选择卡片，商家选择后继续 |

### 3.1 主动下架

推房型选择卡片：

```
┌──────────────────────────────────────┐
│ 要下架哪些房间？                     │
│                                      │
│  [☑ 标准间（3间可售）]               │
│  [☑ 大床房（2间可售）]               │
│  [☑ 全部下架]                        │
└──────────────────────────────────────┘
```

商家确认后调用 API 执行下架，推送下架成功卡片（格式同 §2.2）。

### 3.2 主动上架

推房型选择卡片：

```
┌──────────────────────────────────────┐
│ 要上架哪些房间？                     │
│                                      │
│  [☑ 标准间（当前已下架）]            │
│  [☑ 大床房（当前已下架）]            │
│  [☑ 全部上架]                        │
└──────────────────────────────────────┘
```

商家确认后调用 API 执行上架，推送上架成功卡片：

```
┌──────────────────────────────────────┐
│ ✅ 已恢复上架                        │
│                                      │
│ 房型     大床房                      │
│ 上架时间  05月25日 16:00             │
│                                      │
│           [知道了]                   │
└──────────────────────────────────────┘
```

## 4. 取消订单上架流程

收到平台 API 订单取消通知后立即触发：

1. 读取取消订单对应的房型与房间号
2. 判断该房型当前是否已下架：
   - **已下架** → 立即推上架确认卡片
   - **未下架** → 不推卡片，房间自动回到可售状态，Skill 结束

**上架确认卡片**：

```
┌──────────────────────────────────────┐
│ 📋 有订单取消，房间可重新接单        │
│                                      │
│ 房型     大床房                      │
│ 取消时间  05月25日 15:45             │
│                                      │
│  [立即上架]      [暂不上架]          │
└──────────────────────────────────────┘
```

**按钮响应**：
- `立即上架` → 调用 API 恢复该房型上架，推送上架成功卡片
- `暂不上架` → 关闭卡片，房间维持下架状态，Skill 结束

## 5. 退房上架流程

收到平台 API 退房通知后触发，分两步走：

### 5.1 Step 1 — 推送保洁确认卡片

```
┌──────────────────────────────────────┐
│ 🧹 有客人退房，需要安排保洁          │
│                                      │
│ 房型     标准间                      │
│ 退房时间  05月25日 12:00             │
│                                      │
│ 保洁完成后点击确认，系统将提醒你上架 │
│                                      │
│  [保洁完成 ✓]    [暂不处理]          │
└──────────────────────────────────────┘
```

**按钮响应**：
- `保洁完成 ✓` → 记录保洁完成时间戳，进入 Step 2
- `暂不处理` → 关闭卡片，进入跟进提醒队列（见 §5.3）

### 5.2 Step 2 — 推送上架确认卡片

```
┌──────────────────────────────────────┐
│ ✅ 保洁已完成，可以重新接单了        │
│                                      │
│ 房型     标准间                      │
│ 保洁完成  05月25日 13:30             │
│                                      │
│  [立即上架]      [暂不上架]          │
└──────────────────────────────────────┘
```

**按钮响应**：
- `立即上架` → 调用 API 恢复该房型上架，推送上架成功卡片
- `暂不上架` → 关闭卡片，房间维持下架状态，Skill 结束

### 5.3 保洁跟进提醒

商家点击「暂不处理」后，**2 小时**未标记保洁完成，推送跟进提醒：

```
┌──────────────────────────────────────┐
│ ⏰ 保洁进度跟进                      │
│                                      │
│ 标准间退房已超过 2 小时              │
│ 保洁完成了吗？                       │
│                                      │
│  [保洁完成 ✓]    [还没好]            │
└──────────────────────────────────────┘
```

- `还没好` → 延迟 2 小时再次跟进，最多跟进 3 次后不再主动推送

## 6. Onboarding（首次配置）

### Step 1 — 录入房型信息

```
┌──────────────────────────────────────┐
│ 👋 先来配置你的房间信息              │
│                                      │
│ 房型名称       标准间                │
│ 总房间数       3 间                  │
│                                      │
│  [添加这个房型]   [再加一个房型]     │
└──────────────────────────────────────┘
```

### Step 2 — 触发阈值配置

```
┌──────────────────────────────────────┐
│ 📋 自动下架触发阈值                  │
│                                      │
│ 可售房间数剩余多少间时自动下架？     │
│                                      │
│  [0 间]  [1 间]  [2 间]  [自定义]   │
│                                      │
│ （默认 0 间：完全售罄时才自动下架）  │
└──────────────────────────────────────┘
```

> 各房型共用同一阈值设置，暂不支持按房型单独配置。

## 7. 边界处理速查

| 场景 | 处理方式 |
|------|---------|
| API 下架失败 | 自动重试 1 次；仍失败则推送失败卡片，附 [手动前往榛果商家版] 按钮 |
| API 上架失败 | 同上 |
| 商家主动下架时该房型已下架 | 提示「该房型当前已是下架状态」，不重复操作 |
| 商家主动上架时该房型已上架 | 提示「该房型当前已是上架状态」，不重复操作 |
| 多间同房型同时退房 | 合并为一张保洁确认卡片，标记完成后统一推上架确认 |
| 退房后有新订单立即入住（同房型） | 跳过保洁跟进提醒，不推上架卡片，房间维持当前状态 |
| 保洁跟进超过 3 次仍未完成 | 停止跟进推送，房间维持下架状态，商家可主动发起上架 |
