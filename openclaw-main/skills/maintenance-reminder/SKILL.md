# maintenance_reminder — 客房维修提醒

## 1. 执行入口判断

| 来源 | 判断方式 | 跳转 |
|------|---------|------|
| 上游 Skill 触发 | `hotel_review_manager` 抛出硬件/卫生类差评事件，含房间号/投诉类型/评价摘要 | → §2 生成维修任务（跳过问题读取） |
| 房客 IM 消息 | 美团 IM 消息中识别到维修相关关键词（见 §2.1） | → §2 生成维修任务 |
| 商家主动唤起 | 商家在对话中提及报修/维修/坏了/不好用等意图 | → §2 生成维修任务 |
| 跟进检查 | `trigger_type == "followup_check"`（定时任务，按问题类型间隔触发） | → §4 跟进提醒 |
| 每日汇总写入 | `trigger_type == "daily_summary"` | → §5 日报写入 |

---

## 2. 生成维修任务

### 2.1 问题类型识别

| 问题类型 | 典型关键词 | 跟进超时阈值 |
|----------|-----------|------------|
| 硬件问题 | 空调/热水/网络/门锁/电视/噪音/灯/插座 | 标记「已安排」后 24 小时 |
| 卫生问题 | 脏/头发/异味/床单/毛巾/蟑螂/虫/清洁 | 标记「已安排」后 4 小时 |

> **来源为上游 Skill 触发时**：问题类型由 `hotel_review_manager` 传入，跳过关键词识别。
> **来源为 IM 消息或商家主动时**：读取消息/对话内容，按上表关键词匹配问题类型；无法匹配则归为硬件问题（默认较长处理时效）。

### 2.2 推送维修任务卡片

**硬件问题示例**：

```
┌──────────────────────────────────────┐
│ 🔧 客房维修提醒                      │
│                                      │
│ 问题类型   硬件问题                  │
│ 具体问题   空调不制冷                │
│ 来源       房客差评（5月25日）       │
│                                      │
│ ────────────────────────────────── │
│ 请在 24 小时内完成处理               │
│                                      │
│  [已安排维修]   [暂时无法处理]       │
└──────────────────────────────────────┘
```

**卫生问题示例**：

```
┌──────────────────────────────────────┐
│ 🧹 客房卫生提醒                      │
│                                      │
│ 问题类型   卫生问题                  │
│ 具体问题   浴室发现异味              │
│ 来源       房客 IM 消息（刚刚）      │
│                                      │
│ ────────────────────────────────── │
│ ⚠️ 请在 4 小时内完成处理             │
│                                      │
│  [已安排处理]   [暂时无法处理]       │
└──────────────────────────────────────┘
```

**按钮响应**：
- `已安排维修 / 已安排处理` → 记录安排时间戳，写入跟进队列，推送「已记录」确认卡片，执行 §3
- `暂时无法处理` → 推送「已记录，可在日报中查看待处理任务」提示，记录状态为「待处理」，Skill 暂停

---

## 3. 任务状态跟踪

商家标记「已安排」后，任务进入跟进队列：

```
已安排  →  [超时检查]  →  已完成
              ↓
           超时未完成 → §4 跟进提醒
```

跟进超时阈值：
- 硬件问题：安排后 **24 小时**未标记完成
- 卫生问题：安排后 **4 小时**未标记完成

---

## 4. 跟进提醒

超时未完成时推送跟进卡片：

**硬件问题超时**：

```
┌──────────────────────────────────────┐
│ ⏰ 维修进度跟进                      │
│                                      │
│ 空调不制冷  已安排 24 小时           │
│ 来源  房客差评（5月25日）            │
│                                      │
│ 修好了吗？                           │
│                                      │
│  [已完成 ✓]   [还没好，再等等]       │
└──────────────────────────────────────┘
```

**卫生问题超时**：

```
┌──────────────────────────────────────┐
│ ⚠️ 卫生问题跟进                      │
│                                      │
│ 浴室异味  已安排 4 小时              │
│ 来源  房客 IM 消息                   │
│                                      │
│ 处理好了吗？下一批客人快入住了       │
│                                      │
│  [已完成 ✓]   [还没好，再等等]       │
└──────────────────────────────────────┘
```

**按钮响应**：
- `已完成 ✓` → 记录完成时间戳，任务闭环，写入日报
- `还没好，再等等` → 硬件问题延迟 12 小时再次跟进；卫生问题延迟 2 小时再次跟进；最多跟进 3 次后降级为「长期待处理」状态

---

## 5. 日报写入

每日固定时间将以下数据写入 `hotel_daily_report`：

| 数据项 | 说明 |
|--------|------|
| 当日新增维修任务数 | 当日触发的维修任务总数 |
| 已完成数 | 当日标记完成的任务数 |
| 待处理数 | 未安排或安排后未完成的任务数 |
| 长期待处理数 | 跟进 3 次仍未完成的任务数 |
| 问题类型分布 | 硬件问题 / 卫生问题各自数量 |
| 来源分布 | 差评触发 / IM 触发 / 商家主动各自数量 |

> 写入格式与口径以 `hotel_daily_report` §3 数据项定义为准。

---

## 6. 边界处理速查

| 场景 | 处理方式 |
|------|---------|
| IM 消息关键词无法匹配问题类型 | 归为硬件问题（默认24小时时效），卡片备注「问题类型未能自动识别，请确认」 |
| 同一房间同一问题重复触发 | 合并为同一任务，不重复创建；在原任务卡片追加来源记录 |
| 商家标记完成后同一问题再次触发 | 视为新任务，重新创建并推送 |
| 跟进 3 次仍未完成 | 降级为「长期待处理」，不再主动推送，仅在日报中体现 |
| 日报写入失败 | 本地缓存当日数据，次日合并写入 |

| 维修费用记录与统计 | 待排期 |
