# massage_booking — 按摩预约

## 1. 执行入口判断

**首先判断触发来源，跳转对应分支：**

| 来源 | 判断方式 | 跳转 |
|------|---------|------|
| 定时系统触发（周期复约） | `trigger_type == "scheduled"` + 周期提醒任务存在 | → §5 周期复约提醒 |
| 用户表达负面情绪/疲惫 | 对话中含情绪类信号（见 §2 情绪关怀触发词表） | → §2.1 情绪关怀分支 |
| 用户主动唤起 | 对话中含按摩/推拿/足疗/放松/舒缓等明确意图 | → §2.2 意图识别与分流 |
| 用户未完成配置 | 首次使用，常用地址未设置 | → §6 Onboarding |

---

## 2. 意图识别与分流

### 2.1 情绪关怀分支

**触发词表**（模糊匹配，不要求完全一致）：

| 情绪类型 | 示例表达 |
|---------|--------|
| 疲惫 | "好累"、"累死了"、"累累的"、"整个人都累了"、"身体很重" |
| 压力 | "压力好大"、"好焦虑"、"喘不过气"、"最近太紧绷了" |
| 烦躁 | "好烦"、"心情不好"、"整个人都不好了"、"很烦躁" |
| 身体紧绷 | "肩膀好紧"、"整个人很僵"、"身体很紧"、"需要放松" |

检测到以上信号后，**不直接进入预约流程**，先回应情绪，再轻柔询问：

```
听起来你最近有点累，好好休息一下很重要。
要不要约个按摩放松一下？

[好啊，帮我找找]  [不用，谢谢关心]
```

用户响应分支：

| 用户操作 | 处理 |
|---------|------|
| 点击「好啊，帮我找找」或表示同意 | → §2.2 意图识别与分流 |
| 点击「不用，谢谢关心」或拒绝 | 回应「好的，有需要随时来找我～」，Skill 结束 |
| 追问其他放松方式 | 给出 2–3 条轻量建议（如「也可以约个 SPA」「找个安静的地方坐坐」），不强推预约 |

---

### 2.2 服务意图识别

| 用户意图 | 示例表达 | 服务类型 | 跳转 |
|---------|---------|---------|------|
| 足疗 | "做个足疗""脚很累""足底按摩" | `foot` | → §3 门店查询 |
| 推拿/中式按摩 | "推拿""肩颈按摩""背部推拿""中式按摩" | `tuina` | → §3 门店查询 |
| 泰式按摩 | "泰式按摩""泰式SPA""泰式全身" | `thai` | → §3 门店查询 |
| 全身放松 | "全身按摩""想放松一下""舒缓一下" | `full_body` | → §3 门店查询 |
| 肩颈/局部 | "肩颈不舒服""颈椎疼""腰背酸" | `partial` | → §3 门店查询（标注部位） |
| 意图不明确 | "帮我约个按摩" | — | 追问：「想做足疗、推拿、泰式按摩还是全身放松？」 |

> **局部按摩特殊处理**：意图识别为 `partial` 时，将不适部位（肩颈/腰背/腿部等）作为附加信息传入查询接口，优先推荐有对应专项项目的门店。

---

## 3. 门店查询与推荐

**输入**：服务类型 + 用户地址（GPS/常用地址）+ 不适部位（如有）

**执行步骤**：

1. 调用美团丽人接口，查询附近门店及项目
2. 按以下规则排序，取前 **2 家** 展示，超出折叠：

   | 优先级 | 筛选维度 | 说明 |
   |--------|---------|------|
   | 1 | 历史消费过该门店 | 熟悉环境，用户信任度高 |
   | 2 | 评分 | 4.8分及以上优先 |
   | 3 | 距离 | 同等条件取距离最近 |
   | 4 | 价格 | 同等条件取性价比最高 |

   **冷启动处理**（无历史消费记录）：按综合评分从高到低排列，展示评分 + 服务次数 + 人均价格

3. 无可用门店 → 提示「附近暂无匹配的按摩门店，请尝试调整位置或项目类型」
4. 有结果 → 跳转 §4 预约确认

---

## 4. 预约确认

**执行步骤**：

1. 展示门店推荐卡片（含评分、人均价格、距离）
2. 用户选择门店 → 展示该店可选项目（**含时长选项**，如 60min / 90min / 120min）
3. 用户选择项目及时长 → 展示可选技师（含评分、擅长方向）
4. 用户选择技师（或跳过由系统分配）→ 展示可选时段
5. 用户选择时段 → 展示预约确认卡片
6. 用户确认 → 调用 `order` skill 完成支付 → §4.1 预约结果处理

卡片格式及时长/时段交互详见 [references/booking-flow.md](references/booking-flow.md)

### 4.1 预约结果处理

**成功**：
```
✅ 预约成功！
🧘 [项目名称]（[时长]min） · [门店名称]
👤 技师：[技师姓名] ⭐[评分]
📅 [预约日期] [时间段]
📍 [门店地址]
💰 ¥[金额]
📌 已写入今日行程，明早早报会提醒你出发时间。
```
→ 将预约信息写入当日行程，供 `morning_briefing` 早报「今日行程」模块读取，统一处理叫车提醒
→ 预约成功卡片底部追加轻量收尾文案：「下次想放松一下，随时来找我预约～」，不强制询问周期；周期复约设置入口保留在 §5，用户主动说「帮我设置提醒」时才触发

**失败**：
→ 提示失败原因，保留预约信息，引导重试或换门店/时段

## 5. 周期复约提醒

### 5.1 首次设置（用户主动触发）

用户说「帮我设置按摩提醒」「下次提前提醒我」等时触发，询问：

```
🔄 好的！多久提醒你一次？

[1周后]  [2周后]  [1个月后]  [自定义]
```

用户选择周期后，记录以下信息用于下次复约：
- 服务类型（`foot` / `tuina` / `thai` / `full_body` / `partial`）
- 上次门店信息（优先复用）
- 上次技师信息（优先复用）
- 上次项目及时长信息
- 下次提醒时间 = 当前日期 + 周期 - 1天

### 5.2 到期提醒推送（定时触发）

到期前一天推送复约卡片，格式及交互详见 [references/periodic-reminder.md](references/periodic-reminder.md)

## 7. 边界处理速查

| 场景 | 处理 |
|------|------|
| 附近无可用门店 | 提示调整位置或扩大搜索范围 |
| 用户选择时段已被预约 | 提示「该时段刚被预约」，引导重新选择 |
| 支付失败 | 提示原因，保留预约信息可重试 |
| 门店临时闭店 | 推送通知，引导重新选择门店 |
| 常用地址未设置 | 触发 §6 Onboarding |
| GPS 未授权 | 降级使用 Onboarding 中的常用地址 |
| 用户提及正骨/针灸/理疗 | 明确告知「正骨/针灸等医疗项目暂不支持预约，建议前往正规医疗机构」 |
| 周期提醒到期但用户已手动预约 | 检测到近期已有预约记录则静默跳过 |
| 局部按摩无对应专项项目 | 展示通用放松类项目列表，标注「可告知技师重点关注部位」 |
| 情绪关怀后用户拒绝预约 | 回应「好的，有需要随时来找我～」，不再追问 |
| 情绪信号与明确预约意图同时出现 | 跳过情绪关怀询问，直接进入 §2.2 预约流程 |

## 8. 本期不做（Backlog）

- 上门按摩服务
- 医疗性质的理疗 / 正骨 / 针灸
- 多人同时预约
- 门店会员卡/储值卡管理
- 服务评价/晒图
- 退款/改约流程
- 跨平台比价
