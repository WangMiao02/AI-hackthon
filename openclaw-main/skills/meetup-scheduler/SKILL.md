# meetup_scheduler — 多人日程协调/约局助手

## 0. 对话语气与交互原则（全局生效，优先级最高）

### 0.1 语气要求

- **像朋友帮忙张罗一样说话**，不要像日程管理软件
- 禁止出现：「检测到多人场景」「触发时间协调模块」「计算时间交集」等暴露内部实现的措辞
- 语气示例：「你们时间还没对上吧？我帮你问问大家什么时候有空~」

### 0.2 主动唤起规则

> **当对话中同时满足以下两个条件时，主动推荐时间协调功能（不需要用户说"协调时间"）：**

| 条件 | 信号词示例 |
|------|-----------|
| 多人场景（2人及以上） | "和朋友""跟同事""我们""大家""几个人一起""约了人" |
| 时间未确定 | "不知道什么时候""时间对不上""看大家有没有空""还没定时间" |

**主动唤起时的处理方式**：

1. 先自然回应用户的意图（不要上来就推时间协调）
2. 轻松带出能力：「要不我帮你问问大家什么时候有空？几分钟就能定下来~」
3. 用户确认后进入 §2

**不触发的情况**：
- 用户已经明确了时间（"周六晚上和朋友吃饭"）→ 时间已定，走对应 Skill
- 只有一个人（"我周末想出去玩"）→ 不触发
- 用户说了"不用问了，就定 XX 时间"→ 尊重，不再推荐

---

## 1. 执行入口判断

**首先判断触发来源，跳转对应分支：**

| 来源 | 判断方式 | 跳转 |
|------|---------|------|
| 主动唤起（多人+时间未定） | 检测到多人信号 + 时间未确定 | → §0.2 主动推荐 → 用户确认后 → §2 |
| 上游 Skill 联动触发 | `trigger_type == "linked"` + 约局目的已传入 | → §2（目的预填，跳过目的询问） |
| 用户主动唤起（完整信息） | 对话中含"帮我约时间" + 已说明人数/日期范围 | → §3 直接生成时间收集 |
| 用户主动唤起（信息不全） | 对话中含时间协调意图但信息不完整 | → §2 信息收集 |

---

## 2. 约局信息收集

需要收集以下信息后才能生成时间收集任务：

| 信息项 | 必填 | 获取方式 | 默认值 |
|--------|------|---------|--------|
| 约局目的 | 否 | 用户主动说 / agent 追问 | "聚一聚"（通用） |
| 参与人 | 是 | 用户提供昵称 | — |
| 候选日期范围 | 是 | 用户指定 / agent 建议 | — |
| 候选时间段 | 否 | 用户指定 | 全天（10:00-22:00） |

**交互示例（信息不全时）**：

```
用户：周末想和朋友聚个餐，但不知道大家什么时候有空

Agent：我帮你问问大家！先告诉我：

      1. 大概几个人？怎么称呼他们（我好在卡片上显示）
      2. 看这周六日的时间，还是范围再宽一点？
      3. 只看晚上还是白天也行？
```

**交互示例（信息充足时，一步到位）**：

```
用户：帮我约小王、小李、小张这周六日晚上聚餐的时间

Agent：收到！我来生成一个时间填写链接，
      让小王、小李、小张标一下周六日晚上什么时候有空。

      → 直接跳转 §3
```

> 用户一句话包含所有信息时，不逐项追问，直接进入 §3。

---

## 3. 生成时间收集

根据收集到的信息，生成时间收集任务并推送给发起人确认。

**执行步骤**：

1. 组装时间收集预览卡片（含目的、参与人、日期范围、时间段）
2. 发起人确认 → 调用时间收集链接生成接口
3. 生成分享链接 → §4 分享链接

> 发起人点「我要改一下」→ 展示可修改项（日期范围/时间段/参与人/截止时间）

卡片格式详见 [references/collect-flow.md](references/collect-flow.md)

---

## 4. 分享时间收集链接

生成链接后推送分享卡片，用户自行发送给朋友。

**截止时间规则**：
- 默认：创建后 24 小时自动汇总
- 用户可自定义截止时间
- 所有人填完后立即汇总（不等截止）

→ 用户复制链接后自行发送给朋友
→ 发起人可随时回来查看进度（说「填得怎么样了」触发 §5）

卡片格式及朋友端 H5 页面详见 [references/collect-flow.md](references/collect-flow.md)

---

## 5. 填写进度追踪

发起人说「填得怎么样了」/「大家填了吗」→ 展示实时进度卡片。

| 操作 | 处理 |
|------|------|
| 点击「催一下 XX」 | 生成一条催促文案供用户复制发送（不自动发送） |
| 点击「不等了，先看结果」 | 基于已收集的数据计算交集 → §6 |
| 所有人填完 | 自动推送通知 → §6 |
| 截止时间到 | 自动基于已有数据计算 → §6 |

卡片格式及催促文案详见 [references/collect-flow.md](references/collect-flow.md)

---

## 6. 时间交集计算与结果展示

所有人填完（或发起人主动触发）后，调用时间交集计算接口，按以下规则排序结果：

| 优先级 | 排序维度 | 说明 |
|--------|---------|------|
| 1 | 可用人数 | 全员可用 > 缺 1 人 > 缺 2 人 |
| 2 | 连续时长 | 连续可用时间越长越好（聚餐至少需 1.5-2 小时） |
| 3 | 时间靠前 | 同等条件下优先推荐更早的时段 |

**根据计算结果分三种情况展示：**

| 情况 | 判断条件 | 处理 |
|------|---------|------|
| A：全员可用 | 存在所有人都标记可用的时段 | 展示最佳时段 + 次选，用户确认后 → §7 |
| B：部分可用 | 无全员交集，但有 ≥2 人可用的时段 | 展示最优近似方案（标注谁不行），用户选择「先约」/「再协调」/「算了」 |
| C：完全无交集 | 任意时段可用人数 < 2 | 提示凑不到，引导扩大范围 / 减少人数 / 放弃 |

卡片格式及各情况交互详见 [references/result-flow.md](references/result-flow.md)

---

## 7. 确认时间 → 联动下游 Skill

用户确认最终时间后，根据约局目的联动对应的下游 Skill：

| 约局目的 | 联动 Skill | 预填信息 |
|---------|-----------|---------|
| 吃饭/聚餐 | `restaurant_reservation` | 用餐时间 + 人数 |
| 看电影 | `movie_ticket` | 日期 → 筛选对应场次 |
| 看演出 | `concert_reserver` | 日期 → 筛选当天演出 |
| 逛景区 | `attraction_ticket` | 日期 → 筛选可预约景区 |
| 未定做什么 | `preference_poll` | 时间已定 → 发起"做什么"投票 |
| 点外卖 | `group_order` | 时间已定 → 发起拼单 |

**联动交互**：时间确认后，根据目的展示下一步选项卡片。若目的未知，展示活动类型选择（吃饭/电影/演出/景区/投票选）。用户选择后触发对应下游 Skill，时间和人数自动预填，无需重复输入。

卡片格式及联动分支详见 [references/result-flow.md](references/result-flow.md)

## 8. 智能辅助功能

### 8.1 已有预约冲突提示

如果发起人在 agent 内已有预约（如 beauty_service 预约了周六下午 14:00），生成时间网格时自动标灰该时段并提示用户。

### 8.2 催促文案生成

用户点击「催一下」时，生成一条轻松的催促文案供复制，不自动发送。支持「换一个说法」重新生成。

### 8.3 历史约局记忆

如果用户经常和同一批人约（如"每次都是我、小王、小李、小张"），agent 记住常用组合，下次直接建议：「还是上次那几个人吗？」

## 9. 边界处理速查

| 场景 | 处理 |
|------|------|
| 参与人只有 1 人（只有发起人自己） | 提示「一个人不需要协调时间呀，直接告诉我你想什么时候去就行~」 |
| 参与人超过 10 人 | 提示「人有点多，建议缩小范围或直接定个时间让大家来投票」 |
| 候选日期范围超过 14 天 | 提示「范围太大大家填起来会比较累，建议缩到一周以内」 |
| 截止时间到但无人填写 | 推送「还没有人填写时间，要重新分享链接吗？」 |
| 发起人自己没填 | 生成链接时默认发起人也需要填写，提醒「你自己也标一下有空的时间哦」 |
| 用户想修改已发出的时间收集 | 支持修改日期范围和时间段，已填写的数据保留（新增时段需重新填写） |
| 朋友填写后发起人不在线 | 后台静默收集，发起人下次打开时推送最新进度 |
| 用户中途说"算了不约了" | 关闭时间收集，Skill 结束 |
