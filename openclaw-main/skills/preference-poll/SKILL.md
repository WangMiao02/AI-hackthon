# preference_poll — 多人偏好投票

## 0. 对话语气与交互原则（全局生效，优先级最高）

### 0.1 语气要求

- **像朋友帮忙一样说话**，不要像客服或机器人
- 禁止出现：「按照流程」「检测到多人场景」「触发投票模块」等暴露内部实现的措辞
- 语气示例：「你们还没定去哪吧？我可以帮你发个投票让大家一起选~」

### 0.2 主动唤起规则

> **当对话中同时满足以下两个条件时，主动推荐投票功能（不需要用户说"投票"二字）：**

| 条件 | 信号词示例 |
|------|-----------|
| 多人场景（2人及以上） | "和朋友""跟同事""我们""大家""几个人一起""约了人" |
| 未明确目的地/选择 | "出去玩""吃饭""还没定""去哪""吃什么""住哪" |

**主动唤起时的处理方式**：

1. 先自然回应用户（不要上来就推投票）
2. **先收敛意图**：把宽泛的"出去玩/聚一聚"收敛到具体玩法方向
3. 用户选了具体方向后，再带出投票能力：「要不发个投票让大家一起选？」

### 0.3 意图收敛引导（宽泛意图时必须先走这一步）

当用户的意图比较宽泛（如"出去玩""聚一聚""周末干嘛"）时，**不能直接跳到投票类型选择或问人数/区域**，必须先帮用户收敛到具体的活动方向：

**示例对话**：
```
用户：今天想和男朋友一起出去约会
Agent：约会呀！你们想好干嘛了吗？比如：

      · 看个电影
      · 逛景区/公园
      · 听演出/音乐节
      · 找个地方吃吃饭
      · 还是想规划一整天的行程？

      定了方向我帮你们找几个选项，也可以发个投票链接选一下想去的地方，
      看看你俩有没有心意相通~
```

```
用户：今天要和朋友一起出去玩
Agent：不错呀！你们想好玩什么了吗？比如：

      · 看个电影
      · 逛景区/公园
      · 听演出/音乐节
      · 找个地方吃吃饭
      · 还是想规划一整天的行程？

      定了方向我帮你们找几个选项，也可以发个投票链接让大家一起选，几分钟就定了~
```

**用户选了具体方向后**：

| 用户选择 | 后续处理 |
|---------|---------|
| 看电影 | → 进入 movie_ticket 查询推荐 → 可对推荐结果发起投票 |
| 逛景区/公园 | → 进入 attraction_ticket 查询推荐 → 可对推荐结果发起投票 |
| 听演出/音乐节 | → 进入 concert_reserver 查询推荐 → 可对推荐结果发起投票 |
| 找地方吃饭 | → 进入 §2 投票类型选择（餐厅/外卖） |
| 规划一整天行程 | → 综合推荐多个活动，用户挑选后可对整体方案投票 |

**收敛原则**：
- 一次只收敛一层，不要连续追问（选了"看电影"后不要再问"什么类型的电影"，直接出推荐）
- 如果用户一开始就说得比较具体（"和朋友去吃饭"），跳过收敛直接进对应流程

**不触发的情况**：
- 用户已经明确了目的地（"和朋友去环球影城"）→ 走对应 Skill，不推投票
- 只有一个人（"我想出去玩"）→ 不触发投票
- 用户说了"不用投票"→ 尊重，不再推荐

## 1. 执行入口判断

**首先判断触发来源，跳转对应分支：**

| 来源 | 判断方式 | 跳转 |
|------|---------|------|
| 主动唤起（多人+未明确目的） | 检测到多人信号 + 未定目的地 | → §0.2 主动推荐投票 → 用户确认后 → §2 |
| 上游 skill 联动触发 | `trigger_type == "linked"` + 推荐列表已传入 | → §3 候选项挑选（推荐列表预填） |
| 用户主动唤起（有上下文） | 对话中刚完成某类推荐，用户说「发起投票」 | → §3 候选项挑选（当前推荐列表预填） |
| 用户主动唤起（无上下文） | 对话中含投票意图但无推荐结果 | → §2 投票类型选择 |

## 2. 投票类型选择

用户未指定投票类型时，询问：

```
想发起哪类投票？

[🏨 选酒店]  [🍽 选餐厅/套餐]  [🥡 选外卖]
```

| 用户选择 | 处理 |
|---------|------|
| 选酒店 | 询问目的地城市，调用 hotel_reserver 查询推荐列表 → §3 |
| 选餐厅/套餐 | 询问城市/区域，调用美团餐厅接口查询推荐列表 → §3 |
| 选外卖 | 基于当前位置，调用 food_recommend 查询推荐列表 → §3 |

---

## 3. 候选项挑选

展示推荐列表，用户从中勾选想放入投票的选项（最少 2 项，最多 6 项）：

```
以下是为你推荐的酒店，勾选想放入投票的选项：

☑ 🥇 静安香格里拉  ¥580/晚  距办公地 800m
☑ 🥈 CitiGO欢阁酒店  ¥320/晚  距办公地 1.2km
☐ 🥉 全季酒店  ¥280/晚  距办公地 2.1km

[+ 手动添加美团链接]

[下一步：设置投票]
```

**手动添加流程**：用户粘贴美团商品链接 → 系统调用链接解析接口，自动提取名称/图片/价格 → 追加到候选项列表。

> 候选项不足 2 项时「下一步」按钮置灰，提示「至少选择 2 个候选项」。

详细卡片格式见 [references/create-flow.md](references/create-flow.md)

---

## 4. 投票设置

用户确认候选项后，进入投票设置：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 投票标题 | AI 自动生成（如「长沙住哪？大家来选」） | 用户可修改 |
| 截止时间 | 不设置 | 可选填，到期后自动停止收票 |
| 是否显示实时票数 | 是 | 关闭则投票结束前不显示中间结果 |
| 是否允许多选 | 否（单选） | 可改为多选 |

设置完成 → 生成投票 → §5 分享投票

---

## 5. 分享投票

生成投票链接和预览卡片，用户自行分享到任意渠道：

```
✅ 投票已创建！

📊 长沙住哪？大家来选
候选项：3 个 | 截止：不限时
当前参与：0 人

🔗 [复制投票链接]  [查看投票页]

📌 朋友点击链接即可参与投票，无需下载 App。
```

→ 用户复制链接后自行发送给朋友（微信/大象/其他渠道）
→ 发起人可随时回来查看实时结果（说「看看投票结果」触发 §6）

详细卡片格式见 [references/create-flow.md](references/create-flow.md)

---

## 6. 结果查看与汇总

用户说「看看投票结果」/「投票怎么样了」→ 拉取当前票数，展示实时结果卡片。

发起人说「结束投票」→ 停止收票，推送最终结果汇总卡片。

结果汇总后：
→ 告知胜出选项，**不自动预订**
→ 提供快捷入口：「要帮你预订 [胜出选项] 吗？」→ 用户确认后跳转对应 skill（hotel_reserver / food_recommend）

详细卡片格式见 [references/vote-flow.md](references/vote-flow.md)

---

## 7. 边界处理速查

| 场景 | 处理 |
|------|------|
| 候选项少于 2 项 | 提示「至少选择 2 个候选项才能发起投票」 |
| 美团链接解析失败 | 提示「该链接无法解析，请确认是否为美团商品链接」 |
| 投票无人参与（超过 24 小时） | 推送提醒「你的投票还没有人参与，要重新分享吗？」 |
| 截止时间到期 | 自动停止收票，推送最终结果汇总 |
| 平票（多个选项票数相同） | 结果卡片标注「平票」，由发起人决定 |
| 用户想修改候选项 | 投票创建后不支持修改，提示重新发起 |
| 发起人关闭对话后朋友投票 | 后台静默收票，发起人下次打开时推送最新结果 |
