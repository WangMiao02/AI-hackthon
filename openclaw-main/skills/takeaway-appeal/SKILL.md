# meituan_review_appeal — 商家差评申诉

## 1. 执行入口判断

| 来源 | 判断方式 | 跳转 |
|------|---------|------|
| 上游 Skill 跳转 | 由 `meituan_review_manager` 差评卡片传入评价内容 | → §2 AI 违规分析（跳过评价读取） |
| 商家主动唤起 | 商家在对话中提及申诉/投诉/违规评价等意图 | → §2 前置检查 |
| 申诉结果回调 | 美团 API 推送申诉结果通知 | → §4 结果处理 |

---

## 2. 前置检查（商家主动唤起时执行）

1. 读取商家配置信息，**配置缺失** → 跳转 §6 Onboarding，Skill 结束
2. 拉取近30日内**未申诉的差评列表**（1-2星），推送选择卡片：

```
┌──────────────────────────────────────┐
│ 📋 选择要申诉的评价                  │
│                                      │
│ ① ⭐☆☆☆☆  5月23日                  │
│    「明显是同行来刷差评的，没有任    │
│      何消费记录……」                  │
│                                      │
│ ② ⭐⭐☆☆☆  5月21日                  │
│    「什么破店，骗子，投诉你们……」    │
│                                      │
│ ③ ⭐☆☆☆☆  5月19日                  │
│    「这家店根本没有我点的菜……」      │
│                                      │
│           [近30日已无差评]           │
└──────────────────────────────────────┘
```

> 商家点击某条评价后，传入评价内容，进入 §3 AI 违规分析。

---

## 3. AI 违规分析

### 3.1 分析评价内容

AI 读取评价文本，从以下美团官方申诉类型中判断最匹配的类型：

| 申诉类型 | 典型特征 |
|----------|---------|
| 与订单无关 | 评价内容与本次订单商品/服务无关联 |
| 疑似同行恶意差评 | 语气异常、内容模板化、无具体消费描述 |
| 内容含侮辱谩骂 | 包含人身攻击、侮辱性词汇 |
| 内容严重失实 | 描述与实际订单信息明显不符 |
| 涉嫌敲诈勒索 | 以差评为要挟索要退款或好处 |

同时生成一段**申诉理由文案**供商家确认使用。

### 3.2 推送申诉确认卡片

```
┌──────────────────────────────────────┐
│ 🔍 AI 申诉分析结果                   │
│                                      │
│ 原始评价                             │
│ 「明显是同行来刷差评的，没有任何     │
│   消费记录……」                        │
│                                      │
│ 建议申诉类型                         │
│ ◉ 疑似同行恶意差评                   │
│ ○ 与订单无关                         │
│ ○ 内容严重失实                       │
│                                      │
│ 申诉理由（可修改）                   │
│ ┌────────────────────────────────┐  │
│ │ 该评价内容与本店订单记录严重不 │  │
│ │ 符，且表述高度模板化，疑似同行 │  │
│ │ 恶意刷差评行为，请平台核实处理 │  │
│ └────────────────────────────────┘  │
│                                      │
│    [确认提交]      [放弃申诉]        │
└──────────────────────────────────────┘
```

**按钮响应**：
- `确认提交` → 执行 §4 提交申诉
- `放弃申诉` → 关闭卡片，Skill 结束

> 商家可在卡片内直接切换申诉类型、修改申诉理由文本，无需重新发起。

---

## 4. 提交申诉

1. 调用美团商家开放平台 API，提交申诉类型 + 申诉理由
2. 记录申诉时间戳、评价 ID、申诉类型
3. 推送提交成功卡片：

```
┌──────────────────────────────────────┐
│ ✅ 申诉已提交                        │
│                                      │
│ 申诉类型   疑似同行恶意差评          │
│ 提交时间   05月23日 14:32            │
│ 预计结果   1-3个工作日内反馈         │
│                                      │
│ 结果到达后将自动通知你               │
│                                      │
│           [知道了]                   │
└──────────────────────────────────────┘
```

4. 将申诉记录写入待跟踪队列，等待美团 API 回调

---

## 5. 结果处理

收到美团 API 申诉结果回调后：

### 5.1 申诉通过

```
┌──────────────────────────────────────┐
│ 🎉 申诉成功                          │
│                                      │
│ 申诉类型   疑似同行恶意差评          │
│ 处理结果   评价已由平台删除          │
│ 结果时间   05月25日 10:15            │
│                                      │
│           [知道了]                   │
└──────────────────────────────────────┘
```

### 5.2 申诉驳回

```
┌──────────────────────────────────────┐
│ ❌ 申诉未通过                        │
│                                      │
│ 申诉类型   疑似同行恶意差评          │
│ 平台反馈   暂未发现违规，维持原评价  │
│ 结果时间   05月25日 10:15            │
│                                      │
│  [再次申诉]       [知道了]           │
└──────────────────────────────────────┘
```

**按钮响应**：
- `再次申诉` → 返回 §3 申诉确认卡片，商家可修改申诉类型或理由后重新提交
- `知道了` → 关闭卡片，Skill 结束

### 5.3 写入日报

无论通过或驳回，均将以下数据写入 `merchant_daily_report`：

| 数据项 | 说明 |
|--------|------|
| 当日申诉提交数 | 当日新发起的申诉总数 |
| 申诉通过数 | 当日收到通过结果的申诉数 |
| 申诉驳回数 | 当日收到驳回结果的申诉数 |
| 待结果申诉数 | 已提交尚未收到结果的申诉数 |

> 写入格式与口径以 `merchant_daily_report` §3 数据项定义为准。


---

## 7. 边界处理速查

| 场景 | 处理方式 |
|------|---------|
| API 提交申诉失败 | 自动重试 1 次；仍失败则推送失败卡片，草稿保留可手动重试 |
| 同一评价重复申诉 | 提示「该评价已有进行中的申诉，请等待结果」，不允许重复提交 |
| 申诉结果超时未返回（超过5个工作日） | 推送超时提示卡片，附 [手动前往美团商家版查询] 按钮 |
| 近30日无可申诉差评 | 推送「近30日暂无可申诉的差评」提示卡片 |
| 驳回后再次申诉内容未修改 | 提示「建议修改申诉理由后再提交，提高通过率」，不强制阻止 |
