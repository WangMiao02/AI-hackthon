# meituan_promotion_advisor — 优惠券/满减方案调整

## 1. 执行入口判断

| 来源 | 判断方式 | 跳转 |
|------|---------|------|
| 后台每日检测 | `trigger_type == "scheduled_check"`（每日营业结束后触发） | → §2 数据检测 |
| 商家主动唤起 | 商家在对话中提及优惠券/满减/活动调整等意图 | → §3 前置检查 |
| 配置缺失 | 首次使用或配置不完整 | → §6 Onboarding |

---

## 2. 数据检测（每日后台静默执行）

### 2.1 拉取数据

从美团商家后台数据接口读取：

| 数据项 | 口径 |
|--------|------|
| 近7日每日销售额 | 到手营业额（扣除平台佣金+骑手佣金） |
| 近3日每日销售额 | 同上 |
| 近7日转化率 | 确认订单页访问量 ÷ 最终下单量 |
| 近3日转化率 | 同上 |
| 当前满减方案 | 门槛金额、减免金额、商家承担比例 |
| 当前折扣券方案 | 券面额、使用门槛、商家承担比例 |

### 2.2 触发判断

同时满足以下任意一条，触发方案推荐：

```
条件 A（销售额下滑）：
近3日销售额均值 < 近7日销售额均值 × 85%

条件 B（转化率下滑）：
近3日转化率均值 < 近7日转化率均值 - 10%
```

> 默认阈值说明：
> - 条件 A：近3日均值较近7日均值下滑超过 15% 触发
> - 条件 B：转化率绝对值下降超过 10 个百分点触发
> - 两个条件独立判断，满足任意一条即触发，触发原因会在卡片中注明
> - 商家可在「活动设置」中修改触发阈值

**未触发**：本日不推送，等待次日检测

**触发**：进入 §4 生成推荐方案

---

## 3. 前置检查（商家主动唤起时执行）

1. 读取商家配置，**配置缺失** → 跳转 §6 Onboarding，Skill 结束
2. **配置完整** → 直接进入 §4 生成推荐方案（跳过触发阈值判断）

---

## 4. 生成推荐方案

### 4.1 方案生成逻辑

基于以下维度综合生成建议调整方案：

| 维度 | 说明 |
|------|------|
| 触发原因 | 销售额下滑 / 转化率下滑 / 商家主动 |
| 当前方案 | 现有满减门槛、折扣券面额 |
| 客单价分布 | 近7日订单客单价区间，确保新门槛设置合理 |
| 商家利润空间 | 基于成本口径（见 §4.2）反推可承受的优惠幅度 |

**调整策略方向**：
- 转化率下滑为主 → 优先降低满减门槛或提高折扣券面额，减少确认页流失
- 销售额下滑为主 → 优先提高满减力度，拉升客单价与订单量
- 两者同时下滑 → 组合调整，门槛与力度同步优化

### 4.2 利润计算口径

单笔订单利润 = 订单金额 - 平台佣金 - 骑手佣金 - 商家承担优惠金额

> 优惠补贴说明：
> - 满减活动：美团与商家按约定比例各承担一部分，计算时仅扣除商家承担部分
> - 折扣券：分为平台券（美团全额承担）和商家券（商家全额承担），仅扣除商家券成本
> - 具体分摊比例从当前活动配置中读取

### 4.3 推送方案对比卡片

```
┌──────────────────────────────────────┐
│ 📉 检测到近期转化率下滑              │
│ 近3日转化率 31%  较上周下降 13%      │
│                                      │
│ ── 满减方案调整建议 ──               │
│                                      │
│          当前方案      建议方案      │
│ 满减门槛  满30减5      满25减5       │
│ 商家承担  ¥3.0/单      ¥3.0/单      │
│                                      │
│ ── 折扣券调整建议 ──                 │
│                                      │
│          当前方案      建议方案      │
│ 折扣券    9折券        8.5折券       │
│ 商家承担  ¥2.5/单      ¥3.2/单      │
│                                      │
│ ── 利润预估 ──                       │
│                                      │
│ 单笔利润  ¥18.5/单  →  ¥17.8/单    │
│ 预计日单量  ↑ 约15%（基于历史数据） │
│ 预计日总利润  ¥2,220  →  ¥2,491    │
│                                      │
│  [全部采纳]  [分别选择]  [暂不调整] │
└──────────────────────────────────────┘
```

**按钮响应**：
- `全部采纳` → 调用 API 同时提交满减 + 折扣券调整，执行 §5
- `分别选择` → 展开分项确认卡片（见下），商家逐项决定
- `暂不调整` → 关闭卡片，记录本次推送已忽略，本周期不再重复推送

## 7. 边界处理速查

| 场景 | 处理方式 |
|------|---------|
| 当前无满减或折扣券活动 | 推送「当前未设置活动，建议新建」卡片，附建议新建方案 |
| API 提交调整失败 | 自动重试 1 次；仍失败则推送失败卡片，附 [手动前往美团商家版] 按钮 |
| 商家忽略推送超过3次 | 降低检测频率为每3日一次，避免持续打扰 |
| 数据接口读取失败 | 跳过本日检测，次日重新执行 |
| 利润预估为负（优惠力度过大） | 在卡片中标红警示「⚠️ 该方案单笔利润为负，请谨慎采纳」，不阻止商家执行 |
| 调整后效果不及预期（次日数据未改善） | 次日检测正常执行，不重复提及上次调整 |
